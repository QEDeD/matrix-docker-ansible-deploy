# Configuring Docker

This playbook installs [Docker](https://www.docker.com/) by default, because all services require it. The default value of `matrix_playbook_docker_installation_enabled` is `true`, so the Docker role runs by default. Docker daemon configuration variables only apply while Docker installation remains enabled in the playbook.

To disable Docker installation and install Docker yourself in another way, use this in your `vars.yml` file:

```yaml
matrix_playbook_docker_installation_enabled: false
```

If you manage Docker yourself, playbook variables for Docker daemon configuration do not apply. Configure Docker manually in that case.

If this server is also managed by another playbook, such as [MASH](https://github.com/mother-of-all-self-hosting/mash-playbook), let only one playbook manage Docker installation and daemon configuration. Both playbooks use the Docker role to write `/etc/docker/daemon.json`, so enabling Docker management in both playbooks means the last playbook run wins and may replace daemon settings from the other playbook. Put the complete desired Docker daemon configuration in one playbook and disable Docker installation in the other. If MDAD should manage Docker, disable Docker installation in MASH by removing `mash_playbook_docker_installation_enabled: true` or setting it to `false`. If MASH should manage Docker, set `matrix_playbook_docker_installation_enabled: false` here and configure the equivalent MASH Docker daemon options there.

## Adjusting Docker's default address pools

Docker automatically chooses IPv4 subnets for newly created bridge networks from its [default address pools](https://docs.docker.com/engine/network/#default-address-pools). A server running many Docker networks can exhaust Docker's built-in pools, which may produce errors like `could not find an available, non-overlapping IPv4 address pool among the defaults to assign to the network`.

Docker's built-in local address pools provide 31 automatically allocated IPv4 subnets:

- 15 `/16` subnets from the `172.17.0.0/16` through `172.31.0.0/16` ranges
- 16 `/20` subnets from the `192.168.0.0/16` range

On a typical Docker installation, the default [`bridge` network](https://docs.docker.com/engine/network/drivers/bridge/) already uses `172.17.0.0/16`, so fewer subnets may be available for new user-defined networks. Other existing Docker networks or host/VPN/LAN routes can reduce the usable count further.

If Docker is managed by the playbook, you can ask the playbook to configure Docker's `default-address-pools` setting by adding this to your `vars.yml` file:

```yaml
matrix_playbook_docker_installation_daemon_options_custom:
  default-address-pools:
    - base: "172.16.0.0/12"
      size: 24
    - base: "192.168.0.0/16"
      size: 24
```

If you already define `matrix_playbook_docker_installation_daemon_options_custom`, merge `default-address-pools` into your existing configuration instead of defining the variable twice.

The playbook builds Docker daemon options by merging its automatic options with `matrix_playbook_docker_installation_daemon_options_custom`. This merge preserves automatic top-level options generated by the playbook and adds your custom keys. If the same key exists in both places, your custom value wins. List values are replaced, not appended, so a custom `default-address-pools` list becomes the complete `default-address-pools` value passed to Docker.

When Docker is managed by the playbook, the resulting Docker daemon options are passed to the Docker role as `docker_daemon_options` and written to `/etc/docker/daemon.json`. This writes the playbook's resulting dictionary; it does not append to an existing daemon configuration file. If you already maintain other Docker daemon settings, include them under `matrix_playbook_docker_installation_daemon_options_custom` together with `default-address-pools`.

With this example, Docker allocates `/24` subnets from the listed private address ranges. The `172.16.0.0/12` range can be divided into 4096 `/24` subnets, and the `192.168.0.0/16` range can be divided into 256 `/24` subnets, for 4352 candidate Docker networks. If the default `bridge` network keeps using `172.17.0.0/16`, Docker cannot use the 256 `/24` subnets inside that `/16`, leaving 4096 candidate networks for new user-defined networks.

Compared to Docker's built-in 31-subnet pool, this `/24` example gives up to 4321 additional candidate networks. Compared to a typical installation where the default `bridge` network already consumes one of the built-in `/16` ranges, this example gives about 4066 additional user-defined networks.

Each `/24` network has up to 254 usable IPv4 addresses, one of which is used by Docker as the bridge gateway.

Choose address ranges that do not overlap with your server's LAN, VPN, or other routed networks. If either example range conflicts with your environment, use your own private address range instead.

This setting affects Docker networks created after the Docker daemon configuration changes. Existing Docker networks keep their current subnets until they are recreated.

Changing Docker daemon options causes the playbook to restart Docker. Treat this as disruptive for running containers and plan it for a maintenance window if your server is already in use. If you are changing this on an existing installation, see [Docker cannot find an available IPv4 address pool](maintenance-and-troubleshooting.md#docker-cannot-find-an-available-ipv4-address-pool) for the troubleshooting workflow that recreates existing Docker networks.
